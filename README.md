## Архитектура и технологический стек

* **Источник данных (Firebird)**: Прокси-БД, хранящая транзакционный журнал обмена (`EXCHANGELOG`), метаданные сообщений (`EGISZ_MESSAGES`), а также справочники лицензий и организаций (`EGISZ_LICENSES`, `JPERSONS`)[cite: 2].
* **ETL и парсер (Python)**: Модуль `egisz_monitor_corp` отвечает за инкрементальную выгрузку, парсинг `MSGTEXT` (SOAP/XML), нормализацию статусов и загрузку фактов в DWH[cite: 2].
* **Оркестратор (Airflow)**: Управление конвейером обработки данных (DAG `egisz_monitor_etl_decoupled`), изоляция этапов выгрузки и обработка перезапусков[cite: 2].
* **Хранилище данных (PostgreSQL)**: Хранение staging-слоя, таблиц фактов и измерений, а также аналитических view (`v_*`, `*_ui`)[cite: 2].
* **BI и визуализация (Metabase)**: Построение оперативных и управленческих дашбордов (как код через `metabase_dashboards/*.json`) с автоматическим провижинингом при старте[cite: 2].
* **Операционный UI (Flask)**: Легкий интерфейс (Config UI) для управления конфигурацией, ручного запуска процессов и контроля healthcheck[cite: 2].

## Логика парсинга и интерпретации данных

Сервис извлекает данные из транспортного уровня логов и преобразует их в бизнес-сущности. Ниже представлена таблица маршрутизации и интерпретации ключевых параметров:

| Сущность | Источник (Firebird / SOAP) | Назначение (PostgreSQL) | Логика интерпретации и трансформации |
| :--- | :--- | :--- | :--- |
| **Идентификатор связи (Fact PK)** | `EXCHANGELOG.MSGTEXT` ➔ тег `<relatesToMessage>` | `fact_remd_processing.relates_to_id` | Главный ключ агрегации. Связывает входящий callback с исходящим запросом клиники. |
| **Локальный ID документа** | XML тег `<localUid>` | `queue_outbound.local_uid_semd` | Приоритетный идентификатор документа. Если тег отсутствует в XML, применяется fallback-логика: используется значение `EGISZ_MESSAGES.DOCUMENTID`. |
| **Регистрационный номер** | XML тег `<emdrId>` | `fact_remd_processing.emdr_id` | Идентификатор СЭМД, присвоенный на стороне федеральной системы РЭМД. |
| **Статус обработки** | XML тег `<status>` | `fact_remd_processing.status` | Текстовые статусы из XML нормализуются в строгий перечислимый тип: `success`, `error`, `unknown`. |
| **Ошибки и отказы** | XML тег `<errors>` | `fact_remd_processing.errors_json` | При статусе `error` парсер извлекает блок ошибок в JSON-массив объектов структуры `[{"code": "...", "message": "..."}]` для детализации на дашбордах. |
| **Идентификатор клиники (JID)** | URL (`LOGTEXT`), `EGISZ_LICENSES`, Карта МО | `dim_clinics.jid` | Консолидируется на лету. При несовпадении данных из транспортного лога и лицензий — в витрину пишется итоговый `jid` и устанавливается маркер расхождения. |
| **Тип документа (СЭМД)** | XML тег `<kind>` | `fact_remd_processing.kind_code` | 2-3 значный код из справочников Минздрава. Если тег отсутствует, код восстанавливается (fallback) из метаданных лицензий и типа сообщения. |

## Шаги выборки, парсинга и сортировки данных

Процесс обработки данных разделён на последовательные логические этапы для обеспечения консистентности и защиты от потерь:

* **Инициализация и кэширование измерений:**
  * Выполняется полная выгрузка справочных таблиц `JPERSONS` и `EGISZ_LICENSES`.
  * Формируется in-memory кэш для быстрого сопоставления JID (идентификатора клиники) с URL доменов из логов обмена.
* **Инкрементальная выборка (Delta Load):**
  * Извлечение сырых логов из `EXCHANGELOG` постранично, используя пагинацию по `LOGID`. Это обеспечивает хронологический порядок.
  * Параллельная выборка связанных метаданных из `EGISZ_MESSAGES` через keyset-пагинацию по `EGMID`.
  * Смещение (offset) сохраняется в таблице `etl_state` (курсоры `last_log_id`, `last_egmid`), что предотвращает повторное чтение одних и тех же записей.
* **Парсинг и нормализация:**
  * Поле `MSGTEXT` (содержащее SOAP-конверт) передается в модуль парсера.
  * Происходит поиск по регулярным выражениям или XML-дереву (теги `relatesToMessage`, `status`, `errors` и др.).
  * Применение fallback-цепочек (например, если нет `localUid`, берем `DOCUMENTID`).
* **Загрузка в Staging и дедупликация:**
  * Разобранные данные помещаются в `staging`-таблицы СУБД PostgreSQL.
  * На уровне БД происходит дедупликация (по `relates_to_id`) и связывание фактов обработки с измерениями времени и клиник.
* **Построение витрин (Materialization & Views):**
  * Обновляются витрины фактов `fact_remd_processing` (для расчёта SLA и процентов ошибок).
  * Пересчитывается очередь зависших документов `queue_outbound` (документы, отправленные в РЭМД, на которые не поступил ответный callback).

## Модель выгрузки и парсинга для DAG Airflow

Для обеспечения отказоустойчивости и мониторинга процесса на уровне инфраструктуры, монолитный скрипт ETL декомпозирован в Airflow DAG (`egisz_monitor_etl_decoupled`)[cite: 2]. Модель работает по следующим принципам:

* **Гранулярность задач (Tasks):** Процесс разбит на независимые таски (`test_connections` ➔ `apply_schema_and_state` ➔ `sync_reference_dictionaries` ➔ `parse_and_sync_journal` ➔ `refresh_outbound`). Это позволяет перезапускать только упавший сегмент пайплайна (например, если отвалилась сеть при загрузке `EXCHANGELOG`, справочники повторно не скачиваются).
* **Управление состоянием (State Management):** Airflow не использует длинные транзакционные блокировки сессий. Вместо этого курсоры чтения (`last_egmid`, `last_log_id`) строго фиксируются в таблице `etl_state` внутри PostgreSQL после успешного завершения таски парсинга.
* **Изоляция исполнения:** Для защиты от конфликтующих запусков по расписанию применяется параметр `max_active_runs=1` на уровне DAG.
* **Передача контекста:** Сложные маппинги (например, словари лицензий `best_lic_by_jid`), собранные на этапе `sync_reference_dictionaries`, инициализируются заново или передаются через промежуточное хранилище в тяжелую задачу `parse_and_sync_journal`.

## Дашборды Metabase (01–06)

Дашборды хранятся как JSON в `metabase_dashboards/`.

Важно про провижининг:

- при `METABASE_FORCE_PROVISION=true` выполняется **полный переимпорт** (коллекция администратора очищается и импортируются все JSON),
- по умолчанию (`METABASE_FORCE_PROVISION=auto`) переимпорт делается **только если** набор JSON изменился или Metabase “видит” неполный набор дашбордов; иначе провижининг пропускается (идемпотентность по SHA набора JSON).

### 01 — Оперативный мониторинг и динамика (`01_operational.json`)

Оперативный дашборд “что происходит сейчас”: распределение статусов (`success`/`error`/`unknown`), срезы по клиникам и типам СЭМД, динамика по времени.

Единица учёта в большинстве карточек — **документ (`relates_to_id`)**, а не строки журнала.

### 02 — Сервис, healthcheck и парсинг журнала (`02_service.json`)

Контроль “здоровья” сервиса и канала данных: сводные сигналы, состояние очереди “без ответа”, тренды по ошибкам канала/парсинга, диагностика staging.

Верхний блок опирается на healthcheck-витрину (`v_health_*_ui`) и отражает состояние за последние 24 часа.

### 03 — Документы без ответа (`03_documents_no_response.json`)

Очередь callback: документы, по которым есть исходящая активность, но callback-факт не найден. Основной инструмент для поиска “зависших” документов и оценки возраста ожидания.

Единица учёта — **`local_uid_semd`**.

### 04 — Ошибки и качество данных (`04_quality_and_errors.json`)

Разбор причин отказов РЭМД и качество данных: топы ошибок, матрицы “тип СЭМД × ошибка”, локализация всплесков по клиникам/типам.

Чтобы топы были корректными, в расчётах используется принцип **“один документ — один отказ”** (без раздувания по количеству элементов в `errors_json`).

### 05 — Управление СЭМД (`05_executive.json`)

Компактный управленческий обзор: объём, доля ошибок, состояние очереди без ответа, ключевые срезы для сравнения периодов.

Напоминание об единицах учёта: факт — `relates_to_id`, очередь — `local_uid_semd`.

### 06 — Архив СЭМД (`06_semd_archive.json`)

Архив для выборок за период и сверок: “плоская” таблица документов с фильтрацией/экспортом, базовые разрезы и тренды.
