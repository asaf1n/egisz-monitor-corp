# EGISZ Monitor Corp

**EGISZ Monitor Corp** — специализированный ETL-сервис для мониторинга интеграции Медицинских Информационных Систем (МИС) с реестрами ЕГИСЗ (в частности, РЭМД). Сервис читает журнал **`EXCHANGELOG`** в Firebird (инкремент по `LOGID`), парсит SOAP из **`MSGTEXT`**, извлекает хост клиники из **`LOGTEXT`**, нормализует данные и загружает витрину в PostgreSQL для визуализации.

### Цели и задачи сервиса
* Автоматизированный сбор логов взаимодействия с ЕГИСЗ из СУБД Firebird.
* Десериализация SOAP-сообщений для извлечения статусов регистрации и идентификаторов документов.
* Формирование аналитических витрин данных в PostgreSQL для контроля качества интеграции.
* Обеспечение техподдержки инструментом для анализа ошибок парсинга и передачи данных.

### Стек технологий
* **Ядро:** Python 3.10+.
* **СУБД Источник:** Firebird (через `firebird-driver`).
* **Хранилище (DWH):** PostgreSQL 15/16.
* **Оркестрация:** Apache Airflow (исполнение DAG по расписанию).
* **Визуализация:** Metabase.
* **Инфраструктура:** Docker, Kubernetes (namespace `egisz-corp`).

### Техническая реализация и логика парсинга

#### Инкрементальная загрузка
Синхронизация выполняется методом инкрементальной выборки. В качестве контрольной точки (watermark) используется поле `EXCHANGELOG.LOGID`, значение которого сохраняется в таблице `etl_state`. Это исключает пропуски при обновлении строк в источнике.

#### Сопоставление данных (Mappings)
Для формирования корректной записи в витрине данных `fact_egisz_transactions` используются следующие правила:

**1. Идентификация документа и корреляция:**
* **Связь с ответом:** Тег `<relatesToMessage>` и остальной SOAP читаются **только** из **`EXCHANGELOG.MSGTEXT`**; в **`LOGTEXT`** — только URL/хост клиники, не XML. Первичный ключ витрины в PostgreSQL — **`relates_to_id`**.
* **localUid СЭМД:** В колонке **`local_uid_semd`** (и во вью `v_egisz_transactions_enriched`) сохраняется значение тега **`<localUid>`** из SOAP, а если его нет — **`EGISZ_MESSAGES.DOCUMENTID`**. По этому полю удобно искать в Metabase; оно согласовано с `localUid` интеграции.
* **Связь с клиникой по сообщению:** `EGISZ_MESSAGES.DOCUMENTID` связывает запись журнала с сообщением; **`REPLYTO`** сопоставляется с **`EGISZ_LICENSES.MO_DOMEN`**, оттуда **`JID`**. Из **`JPERSONS`** подтягиваются **`JNAME`**, **`JINN`** (ИНН) и **`FIR_OID`** (OID МО, в том же смысле, что **`<organization>`** и **`EGISZ_LICENSES.MO_UID`**).

**2. Идентификация юридического лица (Клиники):**
* **Через URL:** Хост `gost-…infoclinica.lan` извлекается из **`EXCHANGELOG.LOGTEXT`**, при отсутствии — из **`EGISZ_MESSAGES.REPLYTO`**, по шаблону `gost-([a-zA-Z0-9_-]+)\.infoclinica\.lan`.
* **Через OID:** Тег `<organization>` из XML (`MSGTEXT`) сопоставляется с **`EGISZ_LICENSES.MO_UID`** для получения `JID`.
* **Реквизиты клиники в витрине:** По разрешённому **`JID`** — сначала из связки лицензии (**`EGISZ_LICENSES` → `JPERSONS`**), при неполных данных — дозаполнение из **`JPERSONS`** по `JID`. В **`dim_clinics`** пишутся `jname`, **`jinn`**, **`fir_oid`** (и `mo_uid` с лицензии/SOAP); во **`v_egisz_transactions_enriched`** они доступны как **`clinic_name`**, **`clinic_inn`**, **`clinic_mo_oid`**.

**3. Классификация типов документов:**
* Код типа документа (KIND) извлекается из тега `<kind>` в XML (**`MSGTEXT`**) или из `EGISZ_LICENSES.KIND`.
* Наименование типа документа определяется по справочнику НСИ 1.2.643.5.1.13.13.11.1520.

### Инфраструктура и доступ

#### Основные компоненты и порты
| Сервис | Адрес в K8s / Локально | Назначение |
| :--- | :--- | :--- |
| **PostgreSQL** | В кластере: `postgres:5432`; на хосте: **127.0.0.1:30432** (NodePort) / compose: 5433 | Хранение витрин данных. |
| **Metabase** | В кластере: `metabase:3000`; на хосте: **http://127.0.0.1:30300/** (NodePort) | Визуализация дашбордов. |
| **Config UI** (`conf-ui`) | В кластере: `conf-ui:8080`; на хосте: **http://127.0.0.1:30808/** (NodePort) | Редактор YAML-конфигурации. |

#### Конфигурация
Настройки подключений: файл `config/egisz_corp.yaml`, переменная **`EGISZ_CORP_CONFIG`** (путь к YAML), при необходимости **`EGISZ_CORP_POSTGRES_*`** и др. (см. `egisz_monitor_corp/config_loader.py`).
