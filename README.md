## Назначение и стек

Система предназначена для:

- оперативного контроля статусов регистрации СЭМД в РЭМД,
- выявления аномалий (всплеск отказов, рост доли `unknown`, очередь без ответа, ошибки канала/парсинга),
- поддержки разборов “что сломалось: МИС, шлюз, ЕГИСЗ или наш канал данных”.

Архитектура (в общих терминах):

| Слой | Технология | Роль |
|------|------------|------|
| Источник | Firebird | Прокси-БД интеграции: журнал обмена и метаданные сообщений |
| ETL / парсер | Python (`egisz_monitor_corp`) | Постраничная выгрузка, сопоставление, разбор SOAP/XML |
| Витрина | PostgreSQL | staging + факт + представления `v_*` и `*_ui` |
| BI | Metabase | Дашборды как код (`metabase_dashboards/*.json`) |
| Операционное управление | Config UI (Flask) | Настройка синхронизации и healthcheck |

Поток данных:

```text
Firebird → ETL/парсер → PostgreSQL (витрина) → Metabase (дашборды) + Config UI (healthcheck)
```

## Единицы учёта и ключевые идентификаторы

Ключевое правило для отчётов: **в агрегатах считаем документы, а не строки журнала**.

- **Документ в факте callback**: один `relates_to_id` (из `relatesToMessage` в SOAP/XML). По нему считаются `success` / `error` / `unknown`.
- **Документ в очереди “без ответа”**: один `local_uid_semd`. Приоритет: `<localUid>` из XML; если тега нет — fallback на `EGISZ_MESSAGES.DOCUMENTID`.
- **Регистрационный номер РЭМД**: `emdr_id` (из `emdrId` в SOAP/XML), если присутствует.
- **Клиника**: `jid` (числовой идентификатор). Определяется из нескольких источников (URL/домены из логов, лицензии, карта МО), в витрине фиксируется итог и отмечаются расхождения.
- **Тип СЭМД**: `kind_code` (обычно 2–3 символа). Берётся из XML; при отсутствии — применяется fallback на данных лицензий/метаданных сообщения.

## Как отличать: отказ РЭМД vs ошибка канала vs “нет ответа”

Для разборов важно разделять три разных класса проблем:

- **Отказ РЭМД**: документ идентифицирован, построена строка факта, но `status = error`. Детали — в `errors_json`/человекочитаемой “сводке ошибок”.
- **Ошибка канала/парсинга**: факт **не** построен (например, нет связки или не удалось извлечь ключевые поля) — запись в `stg_parse_errors`. Это не “отказ РЭМД”, а проблема формата/сопоставления/канала данных.
- **Очередь “без ответа”**: исходящий документ есть в staging, но соответствующего callback-факта нет — это риск задержки/потери callback, проблем шлюза/ЕГИСЗ или разрыва окна данных.

## Дашборды Metabase (01–06)

Дашборды хранятся как JSON в `metabase_dashboards/`.

Важно про провижининг:

- при `METABASE_FORCE_PROVISION=true` выполняется **полный переимпорт** (коллекция администратора очищается и импортируются все JSON),
- по умолчанию (`METABASE_FORCE_PROVISION=auto`) переимпорт делается **только если** набор JSON изменился или Metabase “видит” неполный набор дашбордов; иначе провижининг пропускается (идемпотентность по SHA набора JSON).

### 01 — Оперативный мониторинг и динамика (`01_operational.json`)

Оперативный дашборд “что происходит сейчас”: распределение статусов (`success`/`error`/`unknown`), срезы по клиникам и типам СЭМД, динамика по времени.

Единица учёта в большинстве карточек — **документ (`relates_to_id`)**, а не строки журнала.

### 02 — Сервис, healthcheck и парсинг журнала (`02_service.json`)

Контроль “здоровья” сервиса и канала данных: сводные сигналы, состояние очереди “без ответа”, тренды по ошибкам канала/парсинга, диагностика staging.

Верхний блок опирается на healthcheck-витрину (`v_health_*_ui`) и отражает состояние за последние 24 часа.

### 03 — Документы без ответа (`03_documents_no_response.json`)

Очередь callback: документы, по которым есть исходящая активность, но callback-факт не найден. Основной инструмент для поиска “зависших” документов и оценки возраста ожидания.

Единица учёта — **`local_uid_semd`**.

### 04 — Ошибки и качество данных (`04_quality_and_errors.json`)

Разбор причин отказов РЭМД и качество данных: топы ошибок, матрицы “тип СЭМД × ошибка”, локализация всплесков по клиникам/типам.

Чтобы топы были корректными, в расчётах используется принцип **“один документ — один отказ”** (без раздувания по количеству элементов в `errors_json`).

### 05 — Управление СЭМД (`05_executive.json`)

Компактный управленческий обзор: объём, доля ошибок, состояние очереди без ответа, ключевые срезы для сравнения периодов.

Напоминание об единицах учёта: факт — `relates_to_id`, очередь — `local_uid_semd`.

### 06 — Архив СЭМД (`06_semd_archive.json`)

Архив для выборок за период и сверок: “плоская” таблица документов с фильтрацией/экспортом, базовые разрезы и тренды.
