# EGISZ Monitor Corp

Внутренняя система **отчётности и мониторинга интеграции клиник с ЕГИСЗ Минздрава (РЭМД)**: извлечение данных из прокси-БД (Firebird), разбор асинхронных SOAP callback, построение витрины PostgreSQL и дашбордов Metabase для поддержки и поиска аномалий.

Цель этой документации — дать сотрудникам поддержки и аналитикам «сквозное» понимание:

- какие данные выгружаются и почему,
- как они сопоставляются (ключи и правила),
- как формируются отчёты и как их интерпретировать,
- где искать причины: отказ РЭМД vs ошибка канала/парсинга vs отсутствие callback.

## Кратко: стек и назначение

- **Источник данных**: Firebird (прокси-БД интеграции) — журнал обмена, метаданные сообщений, лицензии/клиники.
- **ETL и парсер**: Python (`egisz_monitor_corp`) — выгрузка страницами, разбор `MSGTEXT` (SOAP/XML), нормализация статусов, построение фактов и staging.
- **Хранилище и витрина**: PostgreSQL — таблицы факта/измерений, staging, отчётные `v_*` и `*_ui` представления.
- **BI и отчёты**: Metabase — дашборды как код (`metabase_dashboards/*.json`), провижининг при старте.
- **Операционный UI**: Config UI (Flask) — конфиг ETL, ручной запуск синхронизации, healthcheck.

Высокоуровневый поток данных:

```text
Firebird (EXCHANGELOG, EGISZ_MESSAGES, EGISZ_LICENSES, JPERSONS)
  → ETL (страницы, сопоставление по MSGID, разбор SOAP из MSGTEXT)
  → PostgreSQL (fact + staging + v_*_ui)
  → Metabase (01–06 дашборды) + Config UI (healthcheck/управление)
```

## Единицы учёта и ключевые идентификаторы (важно для интерпретации)

В отчётах важно считать **документы**, а не строки журнала.

- **Документ в витрине callback (факт)**: **один `relates_to_id`** (из `relatesToMessage` в SOAP/XML). Это основной ключ для агрегатов «успех/ошибка/unknown».
- **Документ в очереди “без ответа”**: **один `local_uid_semd`**. Приоритет: `<localUid>` из XML, иначе `EGISZ_MESSAGES.DOCUMENTID`.
- **Регистрационный номер РЭМД**: `emdr_id` (из `emdrId` в SOAP/XML).
- **Клиника**: `jid` (числовой идентификатор клиники) — определяется из нескольких источников (URL/домены/лицензии/карта МО), в витрине фиксируется итог и отмечаются расхождения.
- **Тип СЭМД**: `kind_code` (код типа документа) — берётся из XML, а при отсутствии — fallback по данным лицензий/сообщения; отображается как код и наименование.

## Пошагово: извлечение, сопоставление и построение витрины

Этот раздел описывает, как из Firebird получаются строки витрины и почему некоторые документы видны в очереди или в ошибках, но не попадают в факт.

### 0) Настройка окна синхронизации (конфиг)

Ключевой параметр — `etl.sync_window_days` (см. `config/egisz_monitor.example.yaml`):

- `> 0`: выборка в «скользящем окне» по датам (журнал — по `EXCHANGELOG.LOGDATE`, метаданные — по `EGISZ_MESSAGES.CREATEDATE`).
- `= 0`: выборка **по всем данным “за курсором”** без предиката по датам.
- `< 0`: полная синхронизация «с начала» со сбросом курсоров.

При любом режиме курсоры хранятся в PostgreSQL в `etl_state` и обновляются после успешной обработки пачек.

### 1) Выгрузка из Firebird: две независимые ленты данных

ETL работает с двумя потоками данных, которые нужно удерживать согласованными:

1) **Журнал обмена**: `EXCHANGELOG`.
   - Пагинация по `LOGID`.
   - Полезные поля:
     - `LOGID`, `LOGDATE` — порядок и временная привязка строки,
     - `MSGID` — ключ связи с метаданными сообщения,
     - `MSGTEXT` — тело ответа (SOAP/XML) для парсинга callback,
     - `LOGTEXT` — транспортные данные (URL), источник для определения клиники/домена.

2) **Снимок метаданных сообщений**: `EGISZ_MESSAGES`.
   - Пагинация по `EGMID` (keyset), курсор `etl_state.last_egmid`.
   - Полезные поля:
     - `EGMID`, `CREATEDATE` — порядок и время “обработки/создания сообщения”,
     - `MSGID` — ключ связи с журналом,
     - `DOCUMENTID` — идентификатор исходящего документа (fallback для `local_uid_semd`),
     - `REPLYTO` — URL/домен, дополнительный источник для определения клиники.

Почему две ленты: строки `EXCHANGELOG` содержат содержимое callback (XML), но часто не содержат всей “паспортизации” сообщения; `EGISZ_MESSAGES` содержит метаданные и “выходной” идентификатор документа, но не заменяет журнал.

### 2) Сопоставление “журнал ↔ метаданные” (ключ: `MSGID`)

Сопоставление делается по равенству:

```text
EXCHANGELOG.MSGID  ==  EGISZ_MESSAGES.MSGID
```

Во время обработки очередной страницы `EXCHANGELOG` ETL гарантирует, что в PostgreSQL есть нужные метаданные сообщения:

- либо они уже есть в staging-снимке `EGISZ_MESSAGES`,
- либо догружаются **по списку `MSGID`**, встреченному в текущей пачке журнала.

Это критично для корректного построения полей, завязанных на `REPLYTO`/`DOCUMENTID` и на клинику.

### 3) Разбор callback из `MSGTEXT` (SOAP/XML)

Для строк журнала, содержащих callback, из `EXCHANGELOG.MSGTEXT` извлекается SOAP/XML и читаются ключевые элементы:

- **Связь callback с исходящим запросом**: `relatesToMessage` → `relates_to_id` (первичный ключ факта).
- **Идентификация документа**:
  - `<localUid>` → `local_uid_semd` (приоритет),
  - если `<localUid>` отсутствует: `EGISZ_MESSAGES.DOCUMENTID`,
  - дополнительно/параллельно: `emdrId` → `emdr_id`.
- **Результат обработки РЭМД**: `status` → нормализация в `success` / `error` / `unknown`.
- **Ошибки отказа РЭМД**: `<errors>` → `errors_json` (массив `{code, message}`), далее используется в отчётах по отказам.
- **Тип СЭМД**: `kind` → `kind_code` (если нет кода в XML — используется fallback по данным лицензий/сообщения).

Разделение смысла полей источника:

- `MSGTEXT` — содержимое ответа РЭМД (XML) и всё, что из него парсится.
- `LOGTEXT`/`REPLYTO` — транспортные адреса/домены для определения клиники и трассировки окружения.

### 4) Когда создаётся строка факта, а когда — “ошибка канала”

Факт `fact_egisz_transactions` создаётся только если выполнены оба условия:

1) есть `relatesToMessage` (иначе callback нельзя связать с исходящим запросом),
2) документ удаётся идентифицировать хотя бы одним из способов: `<localUid>` **или** непустой `DOCUMENTID` **или** `emdrId`.

Если чего-то из этого нет, событие не попадает в факт и записывается в `stg_parse_errors`.

Важно для поддержки: `stg_parse_errors` — это **ошибка канала данных/парсинга** (битый XML, нет связи, не удалось определить идентификаторы), а `status=error` в факте — это **отказ РЭМД** (документ идентифицирован, но РЭМД вернул ошибку).

### 5) Очередь “без ответа” (отдельный поток от факта)

Очередь строится независимо от факта: берутся **исходящие документы** (staging) и проверяется, что по ним **нет** соответствующего callback-факта.

Единица учёта очереди — **один документ** (`local_uid_semd`). Это ключевой отчёт для диагностики «задержка ЕГИСЗ / сбой доставки callback / разрыв окна данных».

## Наборы данных (что именно используется в отчётности)

Таблица ниже — “словарь” данных: источник → ключи → как применяется в отчётах.

| Набор данных / поле | Откуда берём | Ключи сопоставления | Для чего в отчётах/мониторинге |
|---|---|---|---|
| Журнал обмена: `EXCHANGELOG` (`LOGID`, `LOGDATE`, `MSGID`, `MSGTEXT`, `LOGTEXT`) | Firebird | курсор по `LOGID`, связь по `MSGID` | Парсинг callback, диагностика канала данных, трассировка событий |
| Метаданные сообщений: `EGISZ_MESSAGES` (`EGMID`, `MSGID`, `CREATEDATE`, `DOCUMENTID`, `REPLYTO`) | Firebird | курсор по `EGMID`, связь по `MSGID` | Временная привязка “обработано”, идентификатор исходящего документа, домен/URL клиники |
| Лицензии/МО: `EGISZ_LICENSES` (`JID`, `MO_UID`, `MO_DOMEN`, `KIND`) | Firebird | домен/uid/параметры сообщения | Определение клиники (JID), fallback по типу СЭМД |
| Справочник клиник: `JPERSONS` (`JNAME`, `INN` по `JID`) | Firebird | `JID` | Человекочитаемые подписи клиник, фильтры/топы |
| Факт callback: `fact_egisz_transactions` | PostgreSQL (ETL) | ключ `relates_to_id` | Основная витрина статусов (success/error/unknown), аналитика отказов, тренды |
| Очередь исходящих: `stg_egisz_outbound_documents` → отчёты “без ответа” | PostgreSQL (ETL) | `local_uid_semd` (из `<localUid>` или `DOCUMENTID`) | Очередь callback, возраст ожидания, риски доставки/задержки |
| Ошибки канала: `stg_parse_errors` (+ агрегаты по документу) | PostgreSQL (ETL) | группировка по документу | Диагностика парсинга/связей, сигнализация проблем формата/данных |
| Отчётные представления: `v_*` и `v_*_ui` | PostgreSQL | — | Единый слой для Metabase (русские подписи/типы), стабильные поля для карточек |

## Дашборды Metabase (01–06): назначение, формирование, данные

Дашборды описаны JSON в `metabase_dashboards/`. При старте Metabase выполняется провижининг: целевая коллекция администратора очищается и затем импортируются все `*.json`.

Ниже — как читать каждый дашборд: что он показывает, на каких наборах данных основан и какие вопросы помогает решать.

### 01 — Оперативный мониторинг и динамика (`01_operational.json`)

Оперативный дашборд для смены/поддержки: «что происходит сейчас» и «как меняется динамика». Основные срезы — распределение статусов (`success`/`error`/`unknown`), топы клиник и типов СЭМД по объёму/ошибкам, и быстрые тренды за короткие окна (например, последние часы/дни).

Формирование: агрегаты строятся по витрине callback (единица учёта — документ `relates_to_id`) с группировками по `jid`, `kind_code` и временным полям (“обработано”). Для расшифровки отказов используются `errors_json` и человекочитаемые сводки ошибок из витрины.

Используемые наборы данных: `fact_egisz_transactions` + отчётные `v_*_ui`, справочники клиник (`JPERSONS` через `jid`), типы СЭМД (`kind_code` + наименование).

### 02 — Сервис, healthcheck и парсинг журнала (`02_service.json`)

Дашборд для контроля «здоровья» сервиса мониторинга и канала данных: сигнализация по качеству (рост `error`/`unknown`), состояние очереди “без ответа”, видимость проблем парсинга/сопоставления и диагностические срезы по staging.

Формирование: верхние блоки опираются на healthcheck-витрины (`v_health_*_ui`), где в одну страницу сведены показатели за 24 часа, очередь и статус курсоров. В нижней части — детализация ошибок канала (`stg_parse_errors`) и диагностические разрезы по данным, которые ещё не превратились в факт.

Используемые наборы данных: `v_health_*_ui` (агрегаты за 24ч), `stg_parse_errors` (канал), staging/отчётные представления для очереди и витрины, курсоры ETL (`etl_state` в составе healthcheck).

### 03 — Документы без ответа (`03_documents_no_response.json`)

Дашборд очереди callback: список документов, по которым есть исходящая активность, но не найден соответствующий callback-факт. Это главный инструмент для поиска «зависших» отправок и оценки возраста ожидания.

Формирование: берётся staging исходящих документов (единица учёта — `local_uid_semd`) и рассчитывается отсутствие соответствующего факта, а также возраст ожидания на основе времени отправки/создания. Срезы по клиникам и типам СЭМД позволяют быстро локализовать проблему (одна клиника/один тип/системная задержка).

Используемые наборы данных: `stg_egisz_outbound_documents` + отчётные `v_rpt_documents_no_response*_ui` (очередь), справочники клиник/типов СЭМД для подписей и фильтров.

### 04 — Ошибки и качество данных (`04_quality_and_errors.json`)

Дашборд для разбора причин отказов РЭМД и оценки качества заполнения/сопоставления данных. Здесь обычно отвечают на вопросы «почему выросли ошибки», «какие клиники/типы СЭМД дают основные отказы», «какой текст/код ошибки доминирует».

Формирование: ядро — документы со `status = error` в витрине callback (по `relates_to_id`) с извлечением/нормализацией причин из `errors_json`. Для корректных топов учитывается «один документ — один отказ» (без размножения строк по нескольким error-элементам), а также строятся матрицы СЭМД × тип ошибки для поиска всплесков.

Используемые наборы данных: `fact_egisz_transactions` (`status`, `errors_json`, `kind_code`, `jid`) + функции/представления со “сводкой ошибок”, справочники клиник/типов СЭМД.

### 05 — Управление СЭМД (`05_executive.json`)

Управленческий обзор: компактные показатели по объёму, доле ошибок, состоянию очереди и ключевым срезам. Используется для регулярной отчётности и контроля SLA интеграции.

Формирование: агрегаты по витрине callback (объём/успех/ошибка) + метрики очереди “без ответа” и несколько срезов, пригодных для сравнения периодов. Важно помнить единицы учёта: факты — по `relates_to_id`, очередь — по `local_uid_semd`.

Используемые наборы данных: агрегаты из `v_*_ui` по факту, агрегаты очереди из `v_rpt_*_ui`, справочники для подписей.

### 06 — Архив СЭМД (`06_semd_archive.json`)

Архивный дашборд для выборок «по фильтру» и сверок: итоговые таблицы и диаграммы по документам за период, с возможностью проваливаться в подробные строки. Подходит для аналитики «что было отправлено/обработано» и выгрузок для разборов.

Формирование: отчётные представления архива строятся на витрине, нормализуют поля документа, клинику и тип СЭМД, и дают “плоскую” таблицу для фильтрации/экспорта. Визуализации “тип СЭМД → число документов” опираются на ту же таблицу.

Используемые наборы данных: архивные `v_rpt_semd_archive*_ui` (таблица), справочники клиник/типов для подписей.
