# Первичный набор дашбордов (EGISZ, Metabase)

Каталог **целиком копируется в Docker-образ** (`../metabase/Dockerfile` → `/app/metabase_dashboards/`). Скрипт `../metabase/setup-dashboards.sh` (вызов из `../metabase/provision.sh` при старте пода) читает **все** `*.json` и создаёт дашборды в **корне** личной коллекции администратора. Перед импортом в коллекции удаляются все существовавшие дашборды и сохранённые вопросы (см. `wipe_corp_root_collection` в `setup-dashboards.sh`).

**Эталон «по умолчанию» для новой сборки (редактируйте в репозитории, не в UI):**

- **`01_operational.json`** — *01 Оперативный мониторинг* — срез по витрине, field filter `dwh_date` (см. `docs/METABASE.md`);
- **`09_executive.json`** — *09 Управленческий дашборд* — агрегаты, очередь, ошибки, круговые доли (статусы / ЕГИСЗ / СЭМД).
- **`10_errors_top.json`** — *10 Топы ошибок* — итоги, pie и рейтинги по ошибкам, СЭМД и клиникам, сводная таблица группировок.
- **`11_healthcheck.json`** — *11 Healthcheck интеграции* — сигналы (`v_health_signals`), heatmap клиники × дни, age-buckets очереди, тренд ошибок парсинга по **уникальным документам** (представление `v_stg_parse_errors_by_document` / `document_group_key`), сводка прокси-БД (`v_health_proxy_db`). Источник: `sql/005_healthcheck.sql`. Дополнительно опрашивается Config UI вкладкой Healthcheck (`/api/healthcheck`).

**Агрегаты в SQL карточках** считают **уникальные документы** (колбэк: `COUNT(DISTINCT "Связанное сообщение")` = `relates_to_id`; очередь без ответа: `localUid СЭМД` = `document_id` в staging; ошибки парсинга: `document_group_key` в `v_stg_parse_errors_by_document`), а не сырые строки журнала. Детальные списки (`SELECT v.*`) по-прежнему построчные.

Дубликатов в других каталогах нет: при изменениях **коммитятся** эти JSON; после `docker build` образа Metabase и `rollout` пода в кластер провижнинг снова подтянет их в Metabase. Изменения только в UI без правки JSON при следующем импорте **перезаписываются** из репозитория.

**Имена сохранённых вопросов (карточек):** в коллекции Metabase карточки лежат рядом с дашбордами и раньше могли совпадать по названию между дашбордами (например, «% Ошибок за период» на 01 и 09). Для уникальности и сортировки карточки на дашбордах **02–11** в JSON имеют префикс **`NN ·`** (номер дашборда). Эталон **01** не меняли: там прежние короткие имена.

| Файл | Имя дашборда в Metabase (`name`) |
|------|----------------------------------|
| `01_operational.json` | Оперативный мониторинг |
| `02_service.json` | Сервис интеграции |
| `03_errors.json` | Ошибки и разбор |
| `04_documents_no_response.json` | Документы без ответа |
| `05_trends.json` | Тренды и динамика |
| `06_quality.json` | Качество данных |
| `07_errors_deep.json` | Глубокий анализ ошибок |
| `08_pending_agg.json` | Агрегация ожидающих |
| `09_executive.json` | Управленческий дашборд |
| `10_errors_top.json` | Топы ошибок |
| `11_healthcheck.json` | Healthcheck интеграции |

Провижининг задаёт **во всю ширину** (`width: full`) для всех дашбордов по умолчанию (см. `setup-dashboards.sh` и поле `width` в JSON при необходимости переопределения).
